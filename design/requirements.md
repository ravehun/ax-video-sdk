# 需求整理

## 1. 背景
- 目标：基于 AXERA 官方 MSP samples，设计并实现一个可复用的硬件编解码与图像处理项目，并明确不同芯片的 SDK 接入方式。
- SDK 来源目录：`/home/axera/msp_dir`
- 当前工程目录：`/home/axera/ax_video_sdk`
- 已知条件：编译器 toolchain 已拷贝到当前工程的 `toolchains/` 目录。
- 当前状态：当前工程内还没有业务源码，现阶段先完成需求和构建设计收敛。

## 2. 项目定位
- 项目目标不是直接搬运 sample，而是参考 sample 的 MSP 调用方式，沉淀出当前仓库自己的库或模块化封装。
- 项目主要面向 AXERA 芯片上的多媒体硬件能力封装，优先覆盖：
  - 视频编解码
  - 图片编解码
  - 基础图像处理
- 项目构建入口以 CMake 为主，通过指定芯片类型和 MSP 路径自动完成 SDK 头文件、库目录和 toolchain 接入。
- 设计原则：
  - 构建入口统一
  - 平台实现分离
  - `ax620e` 与 `ax650` 单独实现、单独维护

- AX620E 侧已确认的参考 sample：
  - `sample/vdec`：视频/图片解码参考
  - `sample/venc`：视频编码与 JPEG 编码参考

- AX650 侧 sample 目录结构与 AX620E 高度相似，但 API 细节和实现差异需要允许分开落地，不能强制收敛为同一套底层实现。

## 8. 需求目标
- 在当前工程内统一接入 AX620E 系列与 AX650 的 MSP SDK。
- 构建时可通过 CMake 指定 `chip_type` 和 `msp` 路径，并自动选择正确的：
  - 头文件目录
  - 库目录
  - toolchain 文件
- 业务层不需要手工处理 SDK 目录差异。
- 对 AX620E 系列，需要支持同一套代码面向 3 个不同目标平台编译。
- 对 AX650，需要支持单独的 aarch64 MSP 接入方式。
- 项目代码组织要围绕“硬件编解码库 + 图像处理库”的目标展开，而不是 sample 工程的演示式组织方式。
- `ax620e` 与 `ax650` 的功能模块允许分别实现，分别维护，只要求对外构建方式和接口风格尽量一致。

## 9. 构建输入与配置抽象
| 配置项 | 说明 |
| --- | --- |
| `chip_type` | 目标芯片，例如 `ax630c` / `ax620q` / `ax620qp` / `ax650` |
| `msp_path` | MSP 根目录 |
| `AX_TARGET_SOC` | 内部归一化后的目标芯片标识 |
| `AX_MSP_ROOT` | 内部归一化后的 MSP 根目录 |
| `AX_MSP_INCLUDE_DIR` | MSP 头文件目录 |
| `AX_MSP_LIB_DIR` | MSP 库目录 |
| `CMAKE_TOOLCHAIN_FILE` | 对应交叉编译 toolchain |

### 9.1 预期 CMake 使用方式
```bash
cmake -S . -B build \
  -Dchip_type=ax630c \
  -Dmsp_path=/home/axera/msp_dir/ax620e/msp \
  -DCMAKE_TOOLCHAIN_FILE=toolchains/aarch64-none-linux-gnu.toolchain.cmake
```

AXCL 示例：

```bash
cmake -S . -B build_axcl \
  -DAXSDK_CHIP_TYPE=axcl \
  -DAXSDK_AXCL_DIR=/usr
```

### 9.2 自动处理逻辑
- 输入 `chip_type` 后，CMake 自动判断目标属于：
  - AX620E 系列
  - AX650
- 输入 `msp_path` 后，CMake 自动推导：
  - include 目录
  - lib 目录
  - 目标平台对应的 ABI 差异
- `CMAKE_TOOLCHAIN_FILE` 由调用方显式传入。
- CMake 负责校验：
  - `chip_type` 与 `msp_path` 是否匹配
  - `chip_type` 与 `CMAKE_TOOLCHAIN_FILE` 是否匹配
  - `msp_path` 是否存在预期目录结构
- 如果上述任一项不匹配，配置阶段直接报错。
- AXCL 作为独立后端处理：
  - 优先使用 `AXSDK_AXCL_DIR`
  - 若未指定，则允许使用系统安装目录如 `/usr/include/axcl`、`/usr/lib/axcl`
  - x86_64 AXCL 默认本机编译，不走交叉编译 toolchain

## 10. 模块拆分建议
| 模块 | 责任 |
| --- | --- |
| `ax_msp_platform` | 处理芯片、SDK、toolchain、include/lib 的映射关系 |
| `ax_common` | 仅放最薄的一层公共类型、错误码、日志和构建辅助，不承载平台特有 MSP 逻辑 |
| `ax_image` | 自定义图像对象，封装格式、分辨率、stride、数据指针与所有权信息 |
| `ax_decoder_dispatch` | 解码后的最新帧缓存、显式获取 API、回调分发、按需拷贝策略 |
| `platform/ax620e` | AX620E 系列独立实现，覆盖视频编解码、JPEG、IVPS 等能力 |
| `platform/ax650` | AX650 独立实现，覆盖视频编解码、JPEG、IVPS 等能力 |
| `platform/ax620e/sample_adapters` | 从 AX620E sample 中抽离的初始化、通道、pool 等逻辑 |
| `platform/ax650/sample_adapters` | 从 AX650 sample 中抽离的初始化、通道、pool 等逻辑 |

### 10.0 对外目录规划
- `include/common`
  - 放公共定义
  - 包括图像类、系统初始化/反初始化 API、公共枚举、错误码、基础配置结构等
- `include/codec`
  - 放编解码器公开定义
  - 包括视频解码器、视频编码器、JPEG 编解码器及相关配置接口
- `include/pipeline`
  - 放后续 pipeline 公开定义
  - 包括 demux、mux、拉流、推流、编解码串联等抽象接口
- `third-party`
  - 放外部依赖
  - 当前推荐放置 RTSP SDK

### 10.1 代码组织原则
- 不要求 `ax620e` 和 `ax650` 共用同一套底层 `.c/.cpp` 实现。
- 公共层只保留真正稳定的内容：
  - 通用数据结构
  - 错误码
  - 日志
  - 构建配置
- 如果某个能力在两个平台上的 API、初始化流程、资源模型不一致，应优先拆成两个平台实现，而不是增加大量条件分支强行复用。

### 10.2 图像对象设计约束
- 需要定义自定义图像类 `ax_image`。
- 第一阶段优先实现 `ax_image`，并要求定义与实现分离。
- 所有编解码模块在涉及原始图像帧输入或输出时，统一使用 `ax_image` 作为数据载体。
- 图像对象应提供静态工厂函数，用于创建指定格式、指定分辨率的图像实例。
- 调用方可以通过静态工厂预先申请目标图像对象，并将其作为显式 API 的出参传入。
- 图像对象至少需要支持以下格式：
  - NV12
  - RGB
  - BGR
- 图像对象的元信息对外只读，包括但不限于：
  - pixel format
  - width
  - height
  - stride
- 外部调用方不应能直接修改图像的分辨率、格式等元信息。
- 图像数据缓冲区如果被外部写入，接口层面难以彻底禁止；因此需求重点是“元信息不可变”，而不是强制“像素数据绝对不可写”。
- `ax_image` 自身需要具备清晰的生命周期管理能力。
- 返回值模式下：
  - 视为调用方更关注易用性而不是极致性能
  - 由内部新申请一张图像并返回给调用方
  - 底层资源通过智能指针等 RAII 机制自动释放
- 出参模式下：
  - 由调用方自己提前申请目标 `ax_image`
  - 调用方自己管理该图像对象生命周期
  - 编解码器或图像处理模块只负责往该对象写入内容，不接管其所有权

### 10.2.1 系统初始化 API
- 需要提供总的系统初始化 API 和系统反初始化 API。
- 该 API 放在 `include/common` 中，作为所有 codec / pipeline 模块的公共入口之一。
- 初始化与反初始化要求具备清晰的调用语义，至少应支持：
  - 初始化成功后再创建和启动各类模块
  - 退出时统一反初始化
  - 重复初始化 / 重复反初始化时行为可控
- 第一阶段建议让该 API 保持轻量，但接口形态先固定下来，便于后续接入 MSP 全局初始化、日志、资源池等逻辑。

### 10.3 解码输出分发模型
- AX 硬件编解码侧的原生图像格式统一按 NV12 处理。
- 解码器需要同时支持两种取帧方式：
  - 显式获取 API
  - 回调通知
- 显式获取 API 至少支持两种形态：
  - 直接返回图像对象，由解码器内部处理底层缓冲的申请、释放和复用
  - 调用方预先创建目标 `ax_image`，再作为参数传入获取指定格式、指定分辨率的图像
- 两种方式都应面向“最新一帧”语义，而不是要求调用方消费每一帧。
- 两种方式都不能阻塞解码线程本身。
- 典型使用方式包括：
  - 业务线程循环 `get frame` 后执行 AI 检测
  - 在回调中直接触发 AI 检测
- 解码线程负责解码与发布最新帧，不负责执行耗时业务逻辑。
- 回调执行应与解码线程解耦，避免用户回调阻塞底层解码流程。
- 数据拷贝时机应尽量延后，只在以下场景发生：
  - 调用方显式获取图像
  - 已注册回调且需要向回调投递图像
- 解码输出图像支持按需指定目标格式和目标分辨率。
- 当调用方请求 RGB/BGR 等非 NV12 格式时，应由解码器内部结合图像处理能力完成 NV12 到目标格式的转换。
- 解码输出图像的底层内存申请、释放与复用由解码器内部负责，外部调用方不承担这部分生命周期管理。

### 10.4 编码输入约束
- 编码器在硬件侧统一使用 NV12 作为输入格式。
- 如果调用方传入的 `ax_image` 不是 NV12，则编码器内部需要负责将其转换为 NV12，或通过图像处理模块完成预处理后再送入编码流程。
- 调用方仍然统一通过 `ax_image` 传递输入图像，不直接感知底层 NV12 约束。

### 10.5 编码输入队列与丢帧策略
- 编码器内部应有独立的输入队列或 pool，用于缓存待编码图像。
- 当编码器面向实时场景工作时，队列打满后不应阻塞生产者线程等待旧帧编码完成。
- 默认策略建议为：`drop_oldest`
  - 当队列已满时，丢弃队列中最旧的待编码输入帧
  - 保留最新输入帧进入编码队列
  - 目标是保证编码输出尽量反映当前最新画面，而不是落后很多帧
- 这里丢弃的对象应限定为“尚未送入硬件编码处理的旧帧”或“仍在软件排队中的旧帧”，不应影响已经开始编码的帧。
- 该策略尤其适合以下场景：
  - 实时预览
  - AI 分析结果录像
  - 对时延敏感、允许丢帧的流式处理
- 如果后续存在录像归档、证据保全等不允许随意丢帧的场景，则应允许扩展其他策略：
  - `block`
  - `drop_newest`
  - `drop_oldest`
- 队列深度应支持配置，例如 `10` 只是默认值，不应写死为固定常量。
- 编码器应提供统计信息，至少包括：
  - 输入帧数
  - 编码成功帧数
  - 丢弃帧数
  - 当前队列深度

### 10.6 Pipeline 拓扑
- `pipeline` 类需要支持：
  - `1 个 demux`
  - `N 个 mux`
- `pipeline` 输入类型不应要求用户手工指定。
- 对外公开配置应尽量只保留 `uri` 等必要字段，输入类型由实现层按 URI 前缀或后缀自动识别，例如：
  - `rtsp://` / `rtsps://` 识别为 RTSP 拉流输入
  - `.mp4` 后缀识别为 MP4 文件输入
- 典型场景包括：
  - 1 路输入，输出主码流 + 子码流
  - 1 路输入，输出录像文件 + RTSP 推流

## 11. 当前已确认的平台边界

### 11.1 AX620E 系列
- `AX630C / AX620Q / AX620QP` 当前统一按 `H.264 decode only` 处理。
- `AX630C / AX620Q / AX620QP` 当前不支持 `H.265 decode`。
- `AX620Q / AX620QP` 的解码能力依赖驱动；若设备上没有 `/soc/ko/ax_vdec.ko`，则用户需要先主动安装驱动，否则解码不可用。
- AX620E 系列不按高并发多路场景设计，现实容量目标以 `1~2 路 1080p` 为主。

### 11.2 AX650
- AX650 功能完整度高于 AX620E 系列，视频编解码、JPEG、IVPS、Pipeline 都作为主要支持目标。
- AX650 与 AX620E 系列保持独立实现和独立维护，不强行做底层统一。

### 11.3 AXCL
- AXCL 作为独立后端实现，不与 AX650 MSP 后端混用，虽然能力上可参考 AX650。
- AXCL 支持单进程多卡，设备通过索引选择；单个 `Pipeline` 仅允许绑定一张卡。
- AXCL 下默认编解码和图像处理都在设备侧完成；`GetLatestFrame` / frame callback 默认返回设备侧图像。
- 用户只有在明确需要 host 访问时，才应显式把图像从设备侧同步到 host。
- AXCL 下当前已经确认稳定的 OSD 能力只有 `rect`。
- `line / polygon / mosaic / bitmap` 在 AXCL 上暂不开放，当前版本应直接拒绝，而不是暴露不稳定实现。
- AXCL 测试和压测应串行运行，避免多个测试进程并行时引入 PCIe/runtime 干扰。

### 12.2 AXCL 已验证结论
- AXCL 本地 smoke 已验证通过的典型链路包括：
  - `ax_transcode_smoke`
  - `ax_pipeline_smoke`
  - `ax_pipeline_frame_smoke`
  - `ax_pipeline_osd_smoke`
- AXCL OSD 当前验证结论为：
  - `rect` 可用
  - 其余绘图类型保持关闭
  - 1 路输入，输出多路不同分辨率或不同码率的编码结果
- `demux` 负责上游输入和解码入口。
- 当 `demux` 输入为 MP4 文件时，后续需要支持循环播放能力；该能力先记为增量需求，不阻塞当前架构实现。
- `mux` 负责各自独立的后处理、编码输出和封装输出。
- 多个 `mux` 应共享同一个上游解码源，但允许各自拥有不同的：
  - 输出分辨率
  - 输出编码参数
  - 输出目标类型

### 10.7 Pipeline 规划
- `pipeline` 是后续阶段要实现的高层能力，用于把 demux、decode、image process、encode、mux 串起来。
- pipeline 层应优先面向场景抽象，而不是暴露底层 MSP 细节。

### 10.8 Demux 能力设想
- 支持输入源：
  - MP4 文件
  - RTSP 拉流
- demux 负责：
  - 打开输入源
  - 自动解析元信息
  - 识别视频编码类型、分辨率、时间基等关键信息
  - 将码流送入解码器
- 对调用方暴露的能力应尽量是“给我一个 URL/文件路径，自动进入可解码状态”。

### 10.9 Mux 能力设想
- 支持输出目标：
  - MP4 写文件
  - RTSP 推流
- mux 本身负责：
  - 接收编码后码流
  - 按输出目标组织封装或发送
- 图像格式适配、分辨率调整、OSD、编码等工作应属于各个输出分支内部的前置处理流程，而不是由 mux 本体直接承担。
- 需要支持自定义输出分辨率。
- 无论输入图像是什么格式，都应先通过硬件图像处理链路转换到编码需要的格式，再进行编码和封装。
- 对 RTSP 推流场景，应支持本地直接起推流服务，不依赖外部第三方 RTSP 服务器。

### 10.9.1 输出分支建议
- 对外虽然可以抽象为 `1 demux + N mux`，但内部实现建议为 `1 demux + N output branches`。
- 每个 output branch 可包含：
  - 图像处理
  - OSD
  - 编码器
  - mux / sink
- 这样主码流、子码流等多路输出更容易独立配置和维护。

### 10.10 OSD 能力设想
- `pipeline` 需要支持 OSD 设置能力。
- 期望支持的 OSD 类型至少包括：
  - 画框
  - 画线
  - 画点
  - 画 mask
- 具体支持范围最终以硬件 API 实际能力为准。
- OSD 设置接口应接收结构化配置，而不是零散参数。
- OSD 设置过程不能阻塞 `demux -> decode -> mux` 主流程。
- OSD 配置应异步生效：
  - 业务线程提交 OSD 配置
  - pipeline 内部在合适时机切换到新的 OSD 配置快照
- OSD 应直接绘制在“最新一帧的解码原图”上，再将绘制后的结果分发给各个 `mux` 做后续处理。
- OSD 生效位置应在多路 `mux` 分发之前，这样同一组 OSD 能同时作用到所有下游输出分支。
- 需要支持 OSD 保留帧数配置：
  - OSD 配置可指定保留多少帧
  - 超过保留帧数后自动失效
  - 有新 OSD 配置进入时，可覆盖旧配置
- 当一次需要设置多个 OSD 元素时，应支持批量提交，并保证它们在同一帧上同时生效。
- 为了保证“同一批 OSD 同时绘制”，内部应以“快照”或“版本号”方式管理 OSD 配置，而不是逐条立即生效。
- OSD 更新与图像绘制之间要避免竞态，不能出现一帧只画出半套 OSD 的情况。

### 10.12 MP4 输入节奏
- MP4 作为输入源时，需要先解析原始时间戳信息以及视频帧率信息。
- 是否按源视频帧率节奏把数据送入解码器，取决于运行模式：
  - `realtime playback` 模式：按源时间戳或帧率节奏推进
  - `offline transcode` 模式：尽可能快地读取和处理，不主动按帧率 sleep
- 第一阶段如果主要服务实时类场景，可以优先支持按源帧率节奏推进的模式。
- 但在设计上不应把“MP4 输入一定按帧率喂给解码器”写死成唯一语义，否则后续离线转码吞吐会被限制。

### 10.11 第三方 RTSP 依赖建议
- 推荐将 RTSP 相关 SDK 放到 `third-party/rtsp-sdk`。
- 候选项目：`https://github.com/ZHEQIUSHUI/rtsp-sdk`
- 当前已确认信息：
  - 仓库存在
  - 仓库定位为 C++ RTSP server/client SDK
- 该依赖适合用于：
  - RTSP 拉流输入
  - RTSP 推流输出
- 基于其 `server/client SDK` 定位，可以推断它适合作为“本地直接起流、不额外依赖第三方 RTSP 服务组件”的候选方案，但具体接口能力仍需后续确认。
- 后续仍需要结合实际接口确认与本项目 `pipeline` 层的封装边界。

## 12. 非功能需求
- 可维护性：芯片、库目录、toolchain 的映射关系应集中维护，避免散落在多个脚本中。
- 可读性：目录和变量命名应直接体现芯片/ABI 差异。
- 可移植性：SDK 根目录建议支持外部配置，不要硬编码为单一绝对路径。
- 稳定性：切换目标芯片时，不应要求改动源码，仅调整构建配置。
- 易用性：调用方优先面向统一库接口，而不是直接使用底层 MSP sample 风格 API。
- 可验证性：每个能力模块后续都应有最小可运行示例或测试程序。
- 一致性：所有编解码模块在图像帧侧使用同一套 `ax_image` 接口模型。
- 非阻塞性：轮询取帧和回调取帧都不能对解码线程形成反压。
- 内存效率：图像数据应尽量延迟拷贝，并优先复用内部缓冲区。
- 生命周期安全：图像对象的所有权边界需要清晰，避免外部误释放或重复释放。
- 生命周期明确：返回值模式和出参模式的所有权规则必须固定且易于理解。
- 接口约束：图像格式、分辨率等元信息创建后不可变。
- 格式转换成本：RGB/BGR 与 NV12 的互转应明确走内部图像处理链路，并关注额外拷贝和转换开销。
- 实时性：编码器在输入过载时应优先保证低时延和最新画面，而不是无限堆积历史帧。
- 分层清晰：公共定义、codec 能力、pipeline 能力和 third-party 依赖要有明确目录边界。
- 第三方隔离：RTSP SDK 应隔离在 `third-party`，避免其头文件和源码直接污染核心模块边界。
- 多路一致性：同一 `pipeline` 内多路 `mux` 对同一帧看到的 OSD 内容应一致。
- 原子性：一批 OSD 更新要么整批生效，要么整批延后到下一帧，不能部分生效。
- 主流程无阻塞：OSD 配置更新不能反向阻塞 demux、decode、encode、mux 主链路。
- 语义清晰：文件输入和实时流输入要允许不同的时间推进策略，避免把“播放”和“转码”语义混为一体。

## 14. 风险与待确认项

### 风险
- AX620E 虽然“逻辑上共用一套头文件”，但实际 SDK 目录中存在 3 份 `include`，后续需要确认是否完全一致。
- AX630C 与 AX650 都使用 aarch64 toolchain，但 MSP 目录结构不同，不能简单复用同一套路径拼接逻辑。
- 如果后续构建系统需要同时支持 sample、msp、第三方库，还需要进一步梳理链接顺序和依赖项。
- sample 偏演示工程，初始化流程、参数组织、日志和错误处理风格可能不适合直接进入正式库，需要二次抽象。
- 视频编解码、JPEG 编解码、IVPS 处理三类能力之间可能共享 buffer/pool/stride 规则，设计不当会导致模块边界重复或耦合过深。
- 如果过度追求跨平台统一实现，后续会在 `ax620e` / `ax650` 差异点上积累大量条件分支，维护成本会迅速升高。
- “最新一帧”语义意味着慢消费者可能拿不到每一帧，后续需要明确定义是否允许丢帧，以及丢帧策略。
- 如果回调直接承载 AI 检测等重任务，需要明确回调执行线程、队列深度和过载时的丢弃策略。
- 由于硬件编解码原生使用 NV12，RGB/BGR 输入输出都依赖额外转换，后续需要评估转换链路对时延和吞吐的影响。
- 如果默认采用 `drop_oldest`，则不适合“每帧必须保留”的归档型编码场景，需要在产品层明确模式边界。
- `pipeline` 一旦同时承担 demux/mux/拉流/推流/编解码编排，边界容易过大，后续需要控制其职责，避免变成大而杂的总控模块。
- OSD 如果直接在共享原图上异步绘制，需要严格处理版本切换和线程安全，否则容易出现半绘制、重复绘制或多路输出不一致。
- OSD 保留帧数语义需要明确定义是“按解码帧计数”还是“按输出帧计数”，否则多路 `mux` 场景下行为会不一致。
- 如果 MP4 输入强制按源帧率节流，则会显著限制离线转码吞吐，因此需要保留两种运行语义。

## 15. 验收标准
| 编号 | 验收项 | 验收标准 |
| --- | --- | --- |
| AC-01 | AX630C 构建配置 | 传入 `chip_type=ax630c` 和有效 `msp_path` 后，能自动选择 `arm64_glibc` 与 `aarch64-none-linux-gnu` toolchain |
| AC-02 | AX620Q 构建配置 | 传入 `chip_type=ax620q` 和有效 `msp_path` 后，能自动选择 `arm_uclibc` 与 `arm-AX620E-linux-uclibcgnueabihf` toolchain |
| AC-03 | AX620QP 构建配置 | 传入 `chip_type=ax620qp` 和有效 `msp_path` 后，能自动选择 `arm_glibc` 与 `arm-linux-gnueabihf` toolchain |
| AC-04 | AX650 构建配置 | 传入 `chip_type=ax650` 和有效 `msp_path` 后，能自动选择 `ax650/msp/out/include`、`ax650/msp/out/lib` 与 aarch64 toolchain |
| AC-05 | 错误校验 | `chip_type` 与 `msp_path` 或 `CMAKE_TOOLCHAIN_FILE` 不匹配时，CMake 配置阶段能直接报错 |
| AC-06 | 视频编解码能力 | 项目具备 H.264/H.265 编码与解码能力封装 |
| AC-07 | JPEG 编解码能力 | 项目具备 JPEG 编码与解码能力封装 |
| AC-08 | 图像处理能力 | 项目具备 crop / resize / csc 基础图像处理能力封装 |
| AC-09 | 平台独立维护 | `ax620e` 与 `ax650` 的实现代码可独立修改，不要求联动改同一份底层逻辑 |
| AC-10 | 配置一致性 | 切换目标芯片时无需修改业务源码，仅调整构建配置 |
| AC-11 | 图像对象统一承载 | 编解码模块在图像帧输入/输出侧统一使用 `ax_image` |
| AC-12 | 图像对象工厂 | 能通过静态工厂创建指定格式、指定分辨率的图像对象 |
| AC-13 | 显式取帧 | 调用方可通过显式 API 获取最新一帧图像，且同时支持“直接返回对象”和“传入目标图像对象作为出参”两种方式，并且不会阻塞解码线程 |
| AC-14 | 回调取帧 | 调用方可通过回调获取最新一帧图像，且回调执行不会阻塞解码线程 |
| AC-15 | 按需拷贝 | 未调用取帧或未注册回调时，不发生额外图像拷贝；发生取帧或回调时才执行拷贝 |
| AC-16 | 输出可配 | 取出的图像支持配置目标格式和目标分辨率，或由调用方通过预分配图像对象显式指定 |
| AC-17 | 生命周期管理 | 直接返回对象模式下，图像内存的申请、释放与复用由解码器内部完成；预分配模式下由图像对象自身管理生命周期 |
| AC-18 | 元信息只读 | 外部不能修改图像对象的分辨率、格式等元信息 |
| AC-19 | 图像格式支持 | `ax_image` 至少支持 NV12、RGB、BGR 三种格式 |
| AC-20 | 编码输入 NV12 适配 | 编码器接收非 NV12 图像时，能够在内部完成到 NV12 的适配后继续编码 |
| AC-21 | 解码输出 NV12 适配 | 解码器默认可输出 NV12；当调用方指定 RGB/BGR 格式时，能够内部完成从 NV12 到目标格式的转换 |
| AC-22 | 编码队列满载行为 | 当编码器输入队列满载时，默认丢弃最旧的待编码帧并保留最新输入帧，不阻塞上游生产者 |
| AC-23 | 编码统计可观测 | 能获取输入帧数、编码成功帧数、丢弃帧数和当前队列深度等统计信息 |
| AC-24 | 公开头文件结构 | 对外接口按 `include/common`、`include/codec`、`include/pipeline` 分层组织 |
| AC-25 | Demux 能力 | 能打开 MP4 或 RTSP 输入，自动解析元信息并进入解码流程 |
| AC-26 | Mux 能力 | 能接收图像或码流并输出 MP4 文件或 RTSP 推流 |
| AC-27 | 输出分辨率控制 | pipeline/mux 流程支持自定义输出分辨率 |
| AC-28 | RTSP 集成方式 | RTSP 相关能力以 `third-party` 独立依赖方式接入，不污染核心 codec 模块 |
| AC-29 | 本地 RTSP 推流 | 在不依赖外部 RTSP 服务器的前提下，支持本地直接推流 |
| AC-30 | Pipeline 多路拓扑 | `pipeline` 支持 `1 个 demux + N 个 mux`，可同时输出主码流、子码流等多路结果 |
| AC-31 | 多 mux 独立输出 | 不同 `mux` 能配置不同的分辨率、编码参数和输出目标 |
| AC-32 | OSD 类型支持 | `pipeline` 支持框、线、点、mask 等 OSD 类型，具体以硬件能力范围内可用为准 |
| AC-33 | OSD 异步设置 | 设置 OSD 结构体时不会阻塞 demux/mux 主流程 |
| AC-34 | OSD 保留与覆盖 | OSD 支持保留帧数控制，并在新配置进入时按既定策略覆盖旧配置 |
| AC-35 | OSD 批量原子生效 | 一次提交的多个 OSD 元素会在同一帧上同时绘制，不出现部分生效 |
| AC-36 | OSD 绘制顺序 | OSD 在最新解码原图上完成绘制后，再统一分发到各个 `mux` 做后续处理 |
| AC-37 | 返回值图像所有权 | 返回值模式下，调用方拿到的是一张可独立持有的 `ax_image`，其底层资源随对象生命周期自动释放 |
| AC-38 | 出参图像所有权 | 出参模式下，调用方自行申请和释放图像对象，接口只负责写入目标内容 |
| AC-39 | MP4 输入节奏 | MP4 输入在设计上支持按源帧率推进的实时语义，并为后续离线高速处理保留扩展空间 |
| AC-40 | 图像类实现方式 | `ax_image` 采用头文件声明、源文件实现的分离方式落地 |
| AC-41 | 系统初始化接口 | 提供可调用的系统初始化和反初始化 API，供后续模块统一接入 |
